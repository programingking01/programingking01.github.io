3.Redis常用命令

  • ZRANGE key 0 -1 返回所有元素，这里的索引 0 代表有序集合中的第一个成员，索引 -1 代表最后一个成员
  • 当在 ZRANGE key 0 -1 后面加上 WITHSCORES 参数后，除了返回有序集合中的成员，还会同时返回每个成员对应的分数。也就是说，结果会以 成员1, 分数1, 成员2, 分数2, ... 这样的形式呈现。
  • ZRANGE 命令的基本语法是 ZRANGE key start stop [WITHSCORES]，当你执行 ZRANGE key 0 -1 或者 ZRANGE key 0 -1 WITHSCORES 时，返回的元素会按照分数从小到大的顺序排列。
  • 如果你需要按照分数从大到小（降序）的顺序获取有序集合中的元素，可以使用 ZREVRANGE 命令。其语法为 ZREVRANGE key start stop [WITHSCORES]。

4.Spring Data Redis

配置引入和初始化

  • RedisConnectionFactory 是创建 RedisConnection 的工厂，通过 RedisConnectionFactory 可以获取 RedisConnection 实例，从而与 Redis 服务器进行交互。RedisConnectionFactory 负责连接的管理和配置，而 RedisConnection 负责具体的命令执行。
  • Redis 模板类RedisTemplate则在 RedisConnectionFactory 和 RedisConnection 的基础上，为开发者提供了方便、高效的操作接口，降低了使用 Redis 的难度。

为什么要设置序列化

  • 在使用 Redis 存储数据时，Java 对象不能直接存储到 Redis 中，因为 Redis 是一个基于内存的键值对存储系统，它只能存储字节数组。所以在将 Java 对象存储到 Redis 之前，需要将其序列化为字节数组；从 Redis 中读取数据时，需要将字节数组反序列化为 Java 对象。

  • 将 RedisTemplate 的键和值的序列化器都设置成了 StringRedisSerializer，目的是保证数据以字符串形式存储，方便阅读和调试。

  • JdkSerializationRedisSerializer 虽然可以处理任意 Java 对象，但它的序列化结果通常比较大，并且序列化和反序列化的性能相对较低。而 StringRedisSerializer 只处理字符串，序列化和反序列化的速度非常快，对于存储简单的字符串类型的数据，使用 StringRedisSerializer 可以显著提高性能。

1. 字符串类型（String）

  • 操作概述：字符串是 Redis 最基本的数据类型，可存储字符串、整数或浮点数。在 Spring Data Redis 里，借助 ValueOperations 进行操作。ValueOperations valueOperations = redisTemplate.opsForValue();

  • 常见操作及对应 Redis 命令

    ：

    • **set(key, value)**：对应 Redis 的 SET 命令，用于设置指定键的值。
    • **get(key)**：对应 Redis 的 GET 命令，用于获取指定键的值。
    • **set(key, value, timeout, unit)**：对应 Redis 的 SETEX 命令，用于设置指定键的值并设置过期时间。
    • **setIfAbsent(key, value)**：对应 Redis 的 SETNX 命令，仅当键不存在时设置键的值。
    • **delete(key)**：对应 Redis 的 DEL 命令，用于删除指定键。
    • increment(key) 方法对指定 key 的值进行自增操作，并返回自增后的结果。-原子性操作，多线程安全，对应 Redis 中的 INCRBY 命令。

2. 哈希类型（Hash）

  • 操作概述：哈希类型用于存储键值对的映射，适合存储对象。在 Spring Data Redis 中，通过 HashOperations 操作。

    HashOperations hashOperations = redisTemplate.opsForHash();

  • 常见操作及对应 Redis 命令

    ：

    • **put(key, hashKey, value)**：对应 Redis 的 HSET 命令，用于设置哈希表中指定字段的值。
    • **get(key, hashKey)**：对应 Redis 的 HGET 命令，用于获取哈希表中指定字段的值。
    • **delete(key, hashKey)**：对应 Redis 的 HDEL 命令，用于删除哈希表中指定字段。
    • **keys(key)**：对应 Redis 的 HKEYS 命令，用于获取哈希表中所有字段。
    • **values(key)**：对应 Redis 的 HVALS 命令，用于获取哈希表中所有值。

3. 列表类型（List）

  • 操作概述：列表是一个有序的字符串列表，可在列表两端进行插入和删除操作。在 Spring Data Redis 中，使用 ListOperations 操作。ListOperations listOperations = redisTemplate.opsForList();

  • 常见操作及对应 Redis 命令

    ：

    • **leftPushAll(key, values)**：对应 Redis 的 LPUSH 命令，用于将一个或多个值插入到列表头部。
    • **leftPush(key, value)**：对应 Redis 的 LPUSH 命令，用于将一个值插入到列表头部。
    • **range(key, start, end)**：对应 Redis 的 LRANGE 命令，用于获取列表中指定范围的元素。
    • **rightPop(key)**：对应 Redis 的 RPOP 命令，用于移除并获取列表的最后一个元素。
    • **size(key)**：对应 Redis 的 LLEN 命令，用于获取列表的长度。

4. 集合类型（Set）

  • 操作概述：集合是无序且唯一的字符串集合。在 Spring Data Redis 中，使用 SetOperations 操作。

    SetOperations setOperations = redisTemplate.opsForSet();

  • 常见操作及对应 Redis 命令

    ：

    • **add(key, values)**：对应 Redis 的 SADD 命令，用于向集合中添加一个或多个成员。
    • **members(key)**：对应 Redis 的 SMEMBERS 命令，用于获取集合中的所有成员。
    • **size(key)**：对应 Redis 的 SCARD 命令，用于获取集合的成员数量。
    • **intersect(key1, key2)**：对应 Redis 的 SINTER 命令，用于获取多个集合的交集。
    • **union(key1, key2)**：对应 Redis 的 SUNION 命令，用于获取多个集合的并集。
    • **remove(key, values)**：对应 Redis 的 SREM 命令，用于移除集合中一个或多个成员。

5. 有序集合类型（ZSet）

  • 操作概述：有序集合和集合类似，不过每个成员都关联了一个分数，可根据分数进行排序。在 Spring Data Redis 中，使用 ZSetOperations 操作。ZSetOperations zSetOperations = redisTemplate.opsForZSet();

  • 常见操作及对应 Redis 命令

    ：

    • **add(key, value, score)**：对应 Redis 的 ZADD 命令，用于向有序集合中添加一个成员并设置分数。
    • **range(key, start, end)**：对应 Redis 的 ZRANGE 命令，用于获取有序集合中指定范围的成员。
    • **incrementScore(key, value, delta)**：对应 Redis 的 ZINCRBY 命令，用于增加有序集合中成员的分数。
    • **remove(key, values)**：对应 Redis 的 ZREM 命令，用于移除有序集合中一个或多个成员。
    • **range(key, start, end)**：对应Redis的ZRANGE命令，按升序获取有序集合里指定范围的成员。结果按分数从小到大排列。
    • reverseRange(key, start, end):对应Redis的ZREVRANGE命令，按降序获取有序集合中指定范围的成员。结果按分数从大到小排列。
    • 索引0标识第一个，-1标识最后一个
    • 索引值为非负整数时，代表从有序集合开头计数的位置。例如，range(key, 0, 2) 会返回分数最小的前 3 个成员。
    • 索引值为负数时，代表从有序集合末尾计数的位置。例如，range(key, -3, -1) 会返回分数最大的后 3 个成员。
    • 若 start 索引大于 end 索引，或者 start 超出有序集合范围，会返回空集合

6.Redis进阶

1.缓存更新策略

  • 总结：
    • 主动更新
    • 先更新数据库，再删除缓存

2.缓存穿透

  • 缓存穿透是指客户端请求的数据在缓存中和数据库中都不存在，这样缓存永远不会生效，这些请求都会打到数据库。

    解决方案：

    • 缓存空对象

      • 优点：简单、方便

      • 缺点：额外的内存消耗、可能造成短期的不一致

    • 布隆过滤

      • 优点：内存占用少，没有多余key
      • 缺点：实现复杂，存在误判问题。

3.缓存雪崩

  • 缓存雪崩是指同一时间大量的缓存key同时失效或者Redis服务宕机，导致大量的请求到达数据库，带来巨大的压力。

  • 解决方案

    • 给不同的key的TTL添加随机值
    • 利用Redis集群提高服务的可用性-主从哨兵
    • 给缓存业务添加限流策略
    • 给业务添加多级缓存

4.缓存击穿

  • 缓存击穿问题也叫热点Key问题，就是一个高并发访问并且缓存重建业务比较复杂的Key突然失效了，无数的请求访问会在瞬间给数据库带来巨大的冲击。

  • 解决方案：

    • 互斥锁

    • 逻辑过期

  • 模拟互斥锁

setIfAbsent() -添加缓存（如果不存在） 返回1-成功，0-失败 redis使用互斥锁实现该机制 -setnx

  • 逻辑过期

添加缓存时，额外添加过期时间字段。

无需处理缓存穿透（缓存未命中直接返回空值/异常）

需要预存带有过期时间的热点数据

存在一段时间的不一致-创建新缓存来代替过期缓存

7.分布式锁-Redisson

可重入锁（Reentrant Lock）

  • 原理：允许同一线程在持有锁的情况下多次获取锁，内部通过计数实现，每次加锁计数 +1 ，解锁计数 -1 ，计数为 0 时才真正释放锁。
  • 应用场景：一个方法调用另一个需要相同锁保护的方法时，避免线程自己阻塞自己。比如递归操作共享资源场景。

公平锁（Fair Lock）

  • 原理：按照请求锁的顺序，先来先得，保证锁分配的公平性。对比非公平锁，它减少了线程 “饥饿” 情况。
  • 应用场景：多个线程对资源访问需求相近，希望公平竞争资源的场景，如多个客户端公平获取数据库连接。

联锁（MultiLock）

  • 原理：将多个锁合并成一个锁，只有当所有子锁都成功获取时，联锁才获取成功；释放时也需释放所有子锁。
  • 应用场景：涉及多个不同资源，需要同时锁定以保证操作原子性的场景，比如同时操作多个不同 Redis 数据库中的数据。

红锁（RedLock）

  • 原理：基于多个 Redis 节点实现，通过在多个节点上获取锁，当在大多数节点（超过半数）成功获取锁时，认为获取到了红锁。
  • 应用场景：对可靠性要求极高，单个 Redis 节点故障也不希望锁机制失效的场景，如金融交易中的分布式事务场景。

读写锁（ReadWriteLock）

  • 原理：区分读操作和写操作，允许多个线程同时读，保证数据一致性；写操作时则独占锁，不允许其他线程读写。
  • 应用场景：读多写少的场景，如缓存系统，大量读请求同时进行，少量写请求更新缓存时需要独占。

信号量（Semaphore）

  • 原理：源自操作系统概念，通过控制信号量值来限制对资源的访问数量，获取信号量时值减 1 ，释放时值加 1 ，值为 0 时线程等待。
  • 应用场景：限制并发访问资源的线程数量，如限制数据库连接池同时使用的连接数。

可过期性信号量（PermitExpirableSemaphore）

  • 原理：在信号量基础上，为获取的许可（permit）设置过期时间，过期后自动释放，无需手动释放。
  • 应用场景：资源使用有时间限制的场景，如限时访问的共享资源。

闭锁（CountDownLatch）

  • 原理：通过计数器控制，初始设定计数，线程调用 countDown() 方法使计数减 1 ，计数为 0 时，等待的线程被释放继续执行。
  • 应用场景：多个线程完成各自任务后，再触发下一步操作的场景，如多个数据加载线程完成数据加载后，通知主线程进行数据处理。

3.联锁（MultiLock）-解决主从一致性

  • 主从模式

    • 主节点执行写
    • 从节点执行读
    • 主节点同步数据至从节点

    • 此时其他线程也能获取锁，出现并发的安全问题
